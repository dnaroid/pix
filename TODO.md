## Разделение логики и представления в pi-ui-extend

### Общая оценка: 7/10 — разделение хорошее, но не идеальное

### ✅ Что работает хорошо

1. Рендеринг хорошо выделен в `rendering/`
- `AppRenderController`, `EditorLayoutRenderer`, `StatusLineRenderer`, `TabLineRenderer`, `ConversationViewport` — все в `src/app/rendering/`
- Рендереры — это чистые view-функции: получают данные → возвращают styled output → не мутируют состояние и не ходят за данными

2. Контроллеры не знают о рендеринге
- Контроллеры из `session/`, `input/`, `commands/` имеют ноль импортов рендеринг-типов (Theme, RenderedLine, ScreenStyler и т.д.)

3. DI-паттерн через host-колбеки
- Каждый контроллер получает `host`-интерфейс (например `AppSessionEventControllerHost`, `AppInputControllerHost`)
- Контроллер вызывает `host.render()` когда нужно обновить экран, но не знает деталей рендеринга

4. `app.ts` — чистый DI-оркестратор
- Конструктор ~640 строк — это wiring (создание контроллеров и проброс зависимостей)
- `render()` — тонкая обёртка, делегирует в `AppRenderController.render()`
- Debounce через `scheduleRender()` с 16ms `setTimeout`

### ⚠️ Проблемы

1. `popup-menu-controller.ts` (~1078 строк) — смешивает логику и представление
- Единственный контроллер, который напрямую импортирует `Theme`, `ScreenStyler`, `colorLine`, `padOrTrimPlain`
- Методы `renderActivePopupMenu()`, `styleOverlayLine()`, `overlayPlainText()` производят `RenderedLine[]` — это задача рендерера, не контроллера
- Рекомендация: вынести рендеринг в `rendering/popup-menu-renderer.ts`

2. Императивный data flow (~70 вызовов `host.render()`)
- Нет reactive store / observable / event bus
- Каждый контроллер вручную дергает `host.render()` после мутации состояния
- ~70 вызовов разбросаны по ~12 контроллерам — работает, но хрупко: легко забыть вызвать render или вызвать лишний раз

3. Shared mutable state через `entries[]`
- Массив `entries[]` принадлежит `app.ts` и передаётся по ссылке в несколько контроллеров
- Мутации через `host.addEntry()` из разных мест — потенциальная гонка состояний

4. Mouse targets как side effect рендеринга
- `AppRenderController.render()` мутирует `mouseController.renderedTargets`, `renderedRowTexts` и т.д.
- Это связывает рендеринг с обработкой ввода — layout и input tracking перемешаны

### 🎯 Приоритетные улучшения

1. Вынести popup-рендеринг из `popup-menu-controller` → `rendering/popup-menu-renderer.ts` (единственное нарушение)
2. Ввести lightweight reactive pattern — хотя бы batch/dirty-flag вместо 70 императивных `host.render()` вызовов
3. Изолировать mouse targets — не мутировать их из рендерера, а возвращать как часть render result

Итого: архитектура в целом чистая для TUI-приложения — рендеринг отделён, контроллеры не знают о визуале. Основные точки роста — popup-controller
и императивный flow рендеринга.
